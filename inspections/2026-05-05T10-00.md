## HIGH Severity Findings

### H-01: Orchestrator is a God class (852 lines)
- **Category**: architecture
- **File**: `src/orchestrator/orchestrator.ts`
- **Description**: The `Orchestrator` class handles dispatching, worker lifecycle, stall reconciliation, tracker reconciliation, retry scheduling, agent event processing, and startup cleanup in a single class. Violates Single Responsibility Principle and makes isolated testing extremely difficult.
- **Suggestion**: Extract into `WorkerRunner`, `Reconciler`, and `EventProcessor` modules. The Orchestrator should remain a thin coordinator.

### H-02: Tight coupling — Orchestrator imports concrete FeishuBitableAdapter
- **Category**: architecture
- **File**: `src/orchestrator/orchestrator.ts` (line 35, 287)
- **Description**: Uses `instanceof FeishuBitableAdapter` to conditionally create MCP servers. Breaks the adapter pattern — adding a new tracker type requires modifying the Orchestrator.
- **Suggestion**: Add `getMcpServerConfig?(issueId: string): McpServerConfig | undefined` to the `TrackerAdapter` interface. Each adapter implements it or returns undefined.

### H-03: No retry limit — infinite retry loops possible
- **Category**: error-handling
- **File**: `src/orchestrator/retry.ts`, `src/orchestrator/orchestrator.ts`
- **Description**: When a worker fails, retries are always scheduled with `attempt + 1`. Attempt count is never checked against a maximum. A permanently broken issue will be retried forever.
- **Suggestion**: Add `max_retry_attempts` config field. Check `attempt >= max_retry_attempts` and transition to a "dead_letter" state.

### H-04: No graceful shutdown for in-flight agent sessions
- **Category**: error-handling
- **File**: `src/cli.ts` (lines 99-105), `src/orchestrator/orchestrator.ts` (lines 71-77)
- **Description**: `orchestrator.stop()` merely clears tick timer and sets `running = false`. In-flight worker promises continue running while `process.exit(0)` is called. Tracker state updates, workspace consistency, and token usage records may be lost.
- **Suggestion**: `stop()` should return `Promise<void>` resolving after all in-flight workers complete or are cancelled. Shutdown handler should `await orchestrator.stop()`.

### H-05: Synchronous file I/O on hot paths blocks event loop
- **Category**: performance
- **File**: `src/logging/execution-log.ts` (line 99), `src/logging/turn-log.ts` (lines 43, 84, 100), `src/metrics/token-log.ts` (line 20)
- **Description**: All log writes use `appendFileSync` on the main thread. `TurnLog.append` is called from `onAgentEvent` on every agent message during streaming. Under high concurrency this blocks the event loop.
- **Suggestion**: Use Bun's `Bun.write` with append mode, or a buffered async writer that batches writes.

### H-06: Path traversal risk in workspace safety validation
- **Category**: error-handling
- **File**: `src/workspace/safety.ts` (lines 8-11)
- **Description**: `validateContainment` uses `resolve()` + `startsWith()` which can be fooled by symlink attacks on certain filesystems.
- **Suggestion**: Use `realpathSync` after `resolve()` to resolve symlinks, with a fallback to string comparison for non-existent paths.

### H-07: Unused import `node:child_process` spawn
- **Category**: maintainability
- **File**: `src/adapters/agent/claude-code/adapter.ts` (line 1)
- **Description**: `import { spawn as nodeSpawn } from "node:child_process"` is never used. The adapter was refactored to use the Agent SDK.
- **Suggestion**: Delete the unused import.

## MEDIUM Severity Findings

### M-01: Duplicate `HooksConfig` type and `expandPath` function
- **Category**: maintainability
- **Files**: `src/workspace/hooks.ts` (line 3), `src/model/workflow.ts` (line 80), `src/workspace/safety.ts` (line 14), `src/workflow/config.ts` (line 52)
- **Description**: Two `HooksConfig` definitions and two `expandPath` implementations that could silently diverge.
- **Suggestion**: Use Zod-inferred type as canonical. Consolidate `expandPath` into `workspace/safety.ts`.

### M-02: `cli.ts` is a 120-line wiring script with no structure
- **Category**: architecture
- **File**: `src/cli.ts`
- **Description**: Entry point does argument parsing, logger setup, adapter registration, config validation, orchestrator construction, TUI setup, signal handling. Repeated `resolvedPath.substring(0, resolvedPath.lastIndexOf("/"))` three times.
- **Suggestion**: Extract `bootstrap()` function or `Application` class. Compute `workflowDir` once.

### M-03: `fetchIssueStatesByIds` fetches ALL records every tick
- **Category**: performance
- **File**: `src/adapters/tracker/feishu-bitable/adapter.ts` (lines 89-98)
- **Description**: Paginates through ALL records then filters client-side by record ID. Called on every tick via `reconcileTrackerStates()`.
- **Suggestion**: Use batch get endpoint or parallel individual fetches via `getRecord(id)`.

### M-04: Tracker tools "get" action fetches ALL records to find one
- **Category**: performance
- **File**: `src/adapters/agent/claude-code/tracker-tools.ts` (lines 62-76)
- **Description**: `listRecords()` fetches all records to find a single record by ID. Should use direct GET endpoint.
- **Suggestion**: Add `getRecord(recordId)` method to `FeishuBitableApi`.

### M-05: `rateLimits` typed as `unknown` throughout
- **Category**: type-safety
- **Files**: `src/orchestrator/state.ts` (line 11), `src/adapters/agent/types.ts` (line 20)
- **Description**: Rate limit data is entirely untyped. `formatRateLimits` does `rateLimits as Record<string, unknown>` without validation.
- **Suggestion**: Define `RateLimitInfo` interface and use it consistently.

### M-06: `AgentSession.metadata` typed as `Record<string, unknown>`
- **Category**: type-safety
- **File**: `src/adapters/agent/types.ts` (lines 27-31)
- **Description**: Forces adapter to cast metadata values when reading (`session.metadata.realSessionId as string`).
- **Suggestion**: Define `AgentSessionMetadata` interface with known fields.

### M-07: FeishuBitable adapter factory uses unsafe type assertions
- **Category**: type-safety
- **File**: `src/adapters/tracker/feishu-bitable/adapter.ts` (lines 130-148)
- **Description**: Every field is cast with `as string` or `as string[]`. Malformed config passes silently and causes runtime errors deep in the adapter.
- **Suggestion**: Use Zod parsing to validate raw config.

### M-08: Monolithic `TrackerConfig` schema
- **Category**: architecture
- **File**: `src/model/workflow.ts` (lines 10-32)
- **Description**: All tracker type fields in one flat schema, no discriminated union on `kind`. No type-safe config per tracker kind.
- **Suggestion**: Use Zod `.discriminatedUnion` on `kind` field, like `workspaceSourceSchema` already does.

### M-09: Liquid template parses twice on every render
- **Category**: performance
- **File**: `src/workflow/prompt.ts` (lines 15-21)
- **Description**: Template is parsed once for validation, then parsed again for rendering.
- **Suggestion**: Parse once, save the parsed template, then render.

### M-10: `canDispatch` creates temporary array per candidate
- **Category**: performance
- **File**: `src/orchestrator/dispatch.ts` (lines 45-47)
- **Description**: `[...state.running.values()].filter(...)` creates a new array for each candidate issue per tick.
- **Suggestion**: Iterate without allocation using a simple loop.

### M-11: Critical reconciliation and retry paths lack tests
- **Category**: complexity
- **Files**: `src/orchestrator/orchestrator.ts`
- **Description**: `reconcileStalled()`, `reconcileTrackerStates()`, `onRetryTimer()` have no test coverage. These are the most complex state mutation paths.
- **Suggestion**: Prioritize tests for these paths before scaling.

### M-12: `require("node:crypto")` in ESM module
- **Category**: maintainability
- **File**: `src/workspace/manager.ts` (line 114)
- **Description**: Uses `require()` in ESM module. Works in Bun but inconsistent with the rest of the codebase.
- **Suggestion**: Replace with `import { createHash } from "node:crypto"` at the top.
